# 分类管理与排序设计

## 背景

当前应用的大类与细分分类写死在 `src/constants/categories.ts` 中，用户只能使用预设选项，无法新增、编辑、删除或调整顺序。这与当前“连续录入”“按实际使用习惯记账”的目标不匹配。

本次设计要把分类从静态常量升级为可持久化、可编辑、可排序的数据模型，并兼容已有账单、趋势、备份恢复和连续录入流程。

## 目标

- 支持大类和细分的新增、编辑、删除
- 预设分类与自定义分类统一对待，预设分类也允许直接编辑或删除
- 支持长按拖拽调整大类顺序
- “保存后自动跳到下一个大类”按用户当前实际排序推进
- 删除分类时不删除历史账单
- 编辑分类名称时，历史账单中的对应名称一起更新
- 备份与恢复同时覆盖账单数据和分类配置

## 非目标

- 本次不做分类颜色自定义
- 本次不做多层级细分，只保留“大类 -> 细分”两级结构
- 本次不做跨设备同步

## 数据设计

### 分类存储

新增两张 SQLite 表：

1. `categories`
   - `id`
   - `name`
   - `color`
   - `sort_order`
   - `created_at`
   - `updated_at`

2. `subcategories`
   - `id`
   - `category_id`
   - `name`
   - `sort_order`
   - `created_at`
   - `updated_at`

### 初始化与迁移

- `src/constants/categories.ts` 继续保留，但角色改为“首次初始化种子数据”
- 数据库初始化时，如果 `categories` 为空，则将预设分类与细分写入数据库
- 记账页、趋势页、概览页后续都从数据库读取分类定义，而不是直接使用常量数组

### 历史账单兼容

`expenses` 表继续保留 `category` 与 `subcategory` 字符串字段，不改成外键依赖。

这样做的原因：

- 删除分类后，历史账单仍能保留原始文案
- 旧账单不依赖“分类记录必须存在”才能显示
- 数据恢复与兼容逻辑更稳

## 核心行为规则

### 新增

- 用户可以新增大类
- 用户可以在某个大类下新增细分
- 新增后立即参与记账页的选项展示与连续录入顺序

### 编辑

- 编辑大类名称时，更新分类表中的名称
- 同时批量更新 `expenses.category`
- 编辑细分名称时，更新细分表中的名称
- 同时批量更新 `expenses.subcategory`

默认语义是“我在修改这个分类本身”，不是“新建一个新分类替代旧分类”。

### 删除

- 删除大类前，先统计有多少历史账单正在使用该大类或其细分
- 删除细分前，先统计有多少历史账单正在使用该细分
- 如果存在引用，弹出确认提示：
  - 删除后不会删除历史账单
  - 历史账单仍显示原有分类文字
  - 该分类只是不再出现在可选项中
- 用户确认后允许删除

### 排序

- 大类支持长按拖拽排序
- 排序结果写入 `categories.sort_order`
- 记账页大类展示顺序、连续录入推进顺序、分类管理页展示顺序都使用这个顺序

本次先只要求大类可拖拽排序。细分继续按各自 `sort_order` 展示，但不强制要求首版支持拖拽。

## 交互设计

### 分类管理入口

在记账页增加“分类管理”入口，位置与现有“账单备份与导入”一致，使用单独按钮触发浮窗。

### 分类管理浮窗

分类管理浮窗包含：

- 大类列表
- 每个大类的编辑与删除入口
- 每个大类下的细分列表
- 新增大类按钮
- 在某个大类下新增细分的入口

交互要求：

- 长按大类项进入拖拽排序
- 点击大类项的编辑按钮可改名
- 点击删除按钮前总是先给出引用提示
- 点击浮窗外区域可以关闭

### 记账页联动

- 记账页大类区域改为读取数据库中的实时分类
- 选中某个大类时，再读取该大类下的细分
- 如果当前已选分类被删除：
  - 自动回退到排序后的第一个可用大类
  - 若该大类没有细分，则 `subcategory` 清空

### 连续录入顺序

- `getNextCategoryStep` 不再依赖常量顺序
- 保存成功后跳到“当前大类在数据库排序中的下一个大类”
- 如果当前已经是最后一个大类，则按现有规则返回概览页

## 备份与恢复

备份文件结构从“仅账单”升级为：

- `entries`
- `categories`
- `subcategories`

恢复时：

- 覆盖恢复：账单和分类配置一起覆盖
- 合并导入：账单继续按 ID 去重；分类配置需要按名称与层级做兼容策略，避免重复插入明显相同的分类

首版合并导入策略采用“名称优先去重”：

- 同名大类视为已存在
- 同一大类下同名细分视为已存在

## 实现拆分建议

1. 数据库层
   - 新增分类与细分表
   - 增加查询、增删改、排序、引用统计、批量改名方法

2. 状态层
   - 在现有账本数据 hook 之外，新增分类数据 hook
   - 统一为记账页和管理浮窗提供实时分类数据

3. 记账页
   - 用数据库分类替换静态常量
   - 改造连续录入顺序来源
   - 增加分类管理入口

4. 分类管理浮窗
   - 分类列表
   - 新增 / 编辑 / 删除
   - 长按拖拽排序

5. 备份恢复
   - 升级备份结构
   - 补充分类配置导入导出逻辑

## 测试要求

至少覆盖以下场景：

- 首次初始化时，预设分类成功写入数据库
- 新增大类 / 细分后，记账页可立即选到
- 编辑大类 / 细分后，历史账单名称同步更新
- 删除仍被历史账单引用的分类时，只删除可选项，不删除历史账单
- 大类排序调整后，记账页展示顺序与连续录入顺序同步变化
- 备份文件包含分类配置
- 恢复后分类结构与账单都能还原

## 风险与约束

- 这是一次“运行时分类源”切换，风险点主要在数据库初始化和旧逻辑残留
- 如果仍有代码路径直接引用 `CATEGORY_DEFINITIONS`，会出现显示顺序与真实数据不一致的问题
- 删除分类但保留历史账单，意味着概览与趋势页在汇总历史分类时要允许“账单里的分类名称不存在于当前分类表”

## 验收标准

- 用户可在应用内完成大类与细分的新增、编辑、删除
- 预设分类与自定义分类在权限上无差别
- 大类可长按调整顺序
- 连续录入按当前排序跳转到下一个大类
- 删除分类不会删历史账单
- 编辑分类会同步改历史账单名称
- 备份恢复后，分类结构与账单记录同时恢复
