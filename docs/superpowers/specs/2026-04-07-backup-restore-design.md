# 月度账本备份与恢复设计

## 背景

当前应用的数据保存在设备本地的 SQLite 数据库中。只要用户卸载应用，Android 通常会一并删除应用沙盒内的数据库文件，因此会导致账本记录丢失。

为了支持版本升级、误卸载兜底、跨手机迁移，需要在应用内提供可由用户主动操作的备份与恢复能力。

## 目标

- 支持将当前账本导出为可恢复的备份文件。
- 支持从备份文件恢复账本数据。
- 导入时提供两种模式：
  - `合并导入`
  - `覆盖恢复`
- 导入前校验备份文件是否合法，避免坏文件污染账本。
- 导入完成后明确反馈导入结果。

## 非目标

- 不做云同步。
- 不做自动定时备份。
- 不直接备份 SQLite 数据库物理文件。
- 不做多用户、多设备实时同步冲突处理。

## 方案对比

### 方案一：JSON 逻辑级备份与恢复

把账本明细从数据库读取出来，导出为 JSON 文件；导入时先解析和校验文件结构，再按选定模式写回数据库。

优点：

- 与数据库底层文件解耦，版本兼容性更强
- 可做结构校验和版本迁移
- 适合跨版本、换手机和误卸载后的恢复
- 文件可读，便于人工检查

缺点：

- 需要自行定义备份格式和恢复语义

### 方案二：CSV 导出 + JSON 恢复

额外提供 CSV 供用户查看，恢复仍以 JSON 为准。

优点：

- CSV 便于用表格软件查看

缺点：

- 恢复仍然必须依赖 JSON
- 当前阶段会增加范围但不提升核心安全性

### 方案三：直接备份数据库文件

直接复制 SQLite 文件，恢复时再写回。

优点：

- 理论上是整库镜像

缺点：

- 与 Expo、Android 路径和数据库状态强耦合
- 升级或表结构变化时脆弱
- 更容易出现导入后打不开或库损坏的风险

## 推荐方案

采用 `JSON 逻辑级备份与恢复`。

原因：

- 当前项目的数据层已经集中在 `src/lib/database.ts` 和 `src/hooks/useLedgerData.ts`，导入导出可以在业务层实现，不需要碰数据库文件本体。
- 这种方式最适合未来迭代，因为后面即使新增字段，也能通过 `schemaVersion` 做兼容。
- 能稳定支持你明确要求的两种恢复语义：`合并导入` 与 `覆盖恢复`。

## 备份文件格式

备份文件使用 JSON，建议命名格式：

- `monthly-ledger-backup-YYYY-MM-DD.json`

顶层结构：

```json
{
  "schemaVersion": 1,
  "appVersion": "1.0.3",
  "exportedAt": "2026-04-07T12:00:00.000Z",
  "entries": [
    {
      "id": "abc123",
      "monthKey": "2026-04",
      "amount": 25.5,
      "category": "饮食",
      "subcategory": "食堂",
      "note": "午饭",
      "createdAt": "2026-04-07T12:00:00.000Z"
    }
  ]
}
```

字段说明：

- `schemaVersion`：备份结构版本，用于未来兼容升级
- `appVersion`：导出时应用版本，仅用于追踪来源
- `exportedAt`：导出时间
- `entries`：账本明细数组

每条 `entry` 必须包含：

- `id`
- `monthKey`
- `amount`
- `category`
- `subcategory`
- `note`
- `createdAt`

## 导出设计

导出流程：

1. 从数据库读取全部账本记录
2. 组装备份对象
3. 写入应用可访问的本地文件
4. 调起系统分享面板，让用户保存到文件管理、聊天工具或云盘

导出失败时提示：

- 文件写入失败
- 系统分享失败

但只要备份文件已经成功生成，就不应把后续分享取消视为导出失败。

## 导入设计

导入流程：

1. 打开系统文件选择器
2. 读取 JSON 文件文本
3. 解析并校验顶层结构
4. 校验每条记录的字段完整性与基础格式
5. 让用户选择恢复模式
6. 执行数据库写入
7. 刷新 UI 并展示结果摘要

### 导入模式一：合并导入

规则：

- 以 `id` 作为唯一标识
- 如果当前数据库中已存在相同 `id`，则跳过
- 只插入当前数据库中不存在的记录

优点：

- 安全，适合日常补回备份
- 不会清空当前数据

限制：

- 如果用户手动改过旧备份文件中的某条记录内容，但 `id` 不变，这次导入不会覆盖本地现有记录

### 导入模式二：覆盖恢复

规则：

- 先清空当前账本记录
- 再导入备份中的全部记录

优点：

- 恢复结果最接近“把某次备份完整还原”

风险控制：

- 执行前必须二次确认
- 文案明确提示“当前账本会被清空”

## 校验与错误处理

导入前必须校验：

- 顶层对象存在且为对象
- `schemaVersion` 为已支持版本
- `entries` 为数组
- 每条记录字段类型正确
- `monthKey` 满足 `YYYY-MM`
- `amount` 为有效数字且大于 0
- `category` 为非空字符串

处理策略：

- 如果整个文件结构非法，直接拒绝导入
- 如果某条记录非法，当前版本建议整体拒绝导入，而不是部分吞掉

这样做的原因是：

- 用户在恢复账本时需要“确定性”
- 部分导入虽然更宽容，但更容易造成“为什么少了几条”的理解成本

## 数据层设计

建议在 `src/lib/database.ts` 中新增：

- `exportAllExpenses`
- `importExpensesMerge`
- `replaceAllExpenses`
- `clearAllExpenses`

同时新增一个独立的备份模块，例如：

- `src/lib/backup.ts`

职责：

- 定义备份文件类型
- 负责备份对象序列化与校验
- 负责导入前解析与验证

这样数据库职责和备份文件职责分离，后面如果备份格式升级，不需要把解析逻辑混进数据库层。

## UI 设计

入口建议先放在现有 `趋势` 页底部，新增一个轻量 `数据管理` 区块，而不是立刻新增第四个 Tab。

区块包含两个主操作：

- `导出备份`
- `导入备份`

导入时交互顺序：

1. 选中文件
2. 通过校验后弹出确认
3. 提供两种操作：
  - `合并导入`
  - `覆盖恢复`

导入结果反馈示例：

- 合并导入成功：新增 `12` 条，跳过 `34` 条重复记录
- 覆盖恢复成功：已清空当前账本，并恢复 `46` 条记录

## 兼容性与未来扩展

通过 `schemaVersion` 预留后续能力：

- 新增字段时做向前兼容
- 未来可以在读取旧版本备份时做迁移映射

后续可扩展但不在本轮实现的项：

- 额外导出 CSV 供人工查看
- 自动提醒用户定期备份
- 接入云盘或远程同步

## 测试策略

优先为纯函数与数据层补测试，覆盖：

- 备份对象生成格式正确
- 合法 JSON 能通过校验
- 缺字段、坏格式、非法 `monthKey` 会被拒绝
- 合并导入时重复 `id` 会被跳过
- 覆盖恢复会先清空再导入

界面侧至少验证：

- 能进入文件选择流程
- 导入前会弹确认
- 完成后会显示结果摘要
