# 月度账本预算设置与统一管理入口设计

## 背景

当前应用的预算能力仍停留在固定常量阶段：[ledgerSummary.ts](/D:/mcat/src/lib/ledgerSummary.ts) 中使用统一的 `2000` 元月预算来计算概览页和趋势页里的预算状态、超支统计和预算基准线。这种做法在预算固定时简单有效，但一旦用户希望按自己的实际情况调整预算，就会立刻暴露限制：

- 用户无法修改默认预算。
- 用户无法为单独月份设置不同预算。
- 所有预算展示都绑定固定常量，无法和用户配置同步。
- 记账页顶部目前把“账单备份与导入”和“分类管理”拆成两个入口，后续再加入预算设置后会进一步分散。

这次改动的目标是补齐预算设置能力，并把预算、分类、备份导入整合为一个统一入口，减少入口分裂，同时保证所有预算展示口径一致。

## 目标

1. 支持全局默认预算设置。
2. 支持按月覆盖预算设置。
3. 让用户在“默认预算”和“单月覆盖预算”之间有明确选择权。
4. 让概览页、趋势页和相关统计都基于“该月实际预算”同步更新。
5. 将预算设置、分类管理、备份与导入整合为一个统一入口。
6. 让备份导出和导入恢复覆盖预算配置，避免换机或恢复后预算丢失。

## 非目标

- 不做按分类预算。
- 不做预算提醒、推送、阈值预警通知。
- 不做云同步或多设备实时同步。
- 不改动现有账单、分类和细分的核心模型。
- 不引入独立“设置页”Tab。

## 方案对比

### 方案 A：全局预算 + 按月覆盖

- 保存一个默认预算。
- 允许为某个月保存一条覆盖预算。
- 某月的实际预算解析规则为：月覆盖预算 > 默认预算 > 内置兜底值。

优点：

- 同时满足“统一预算”和“按月预算”两种诉求。
- 与当前“按月记账、按月复盘”的数据结构最契合。
- 预算计算口径稳定，页面同步逻辑清晰。

缺点：

- 数据模型会比当前常量预算多一层预算配置。

### 方案 B：预算模式二选一

- 用户在“全局预算模式”和“按月预算模式”之间切换。
- 两种模式互斥。

优点：

- 概念表面上更直接。

缺点：

- 模式切换会带来迁移和解释成本。
- UI 和数据逻辑都会变复杂。

### 方案 C：全局预算 + 当前月临时预算

- 保存一个默认预算。
- 仅支持当前正在查看月份的临时调整。

优点：

- 改动最小。

缺点：

- 不是真正的按月预算。
- 用户离开当前月后预算心智不清晰。

## 结论

采用方案 A：`全局预算 + 按月覆盖`。

这是本次改动的推荐方案，因为它在心智模型、实现复杂度和后续扩展性之间最均衡。用户既可以只维护一个默认预算，也可以在个别月份做覆盖，而展示层始终只关心“这个月实际使用哪一个预算”。

## 预算数据模型

### 新增预算配置

新增一组预算配置存储，而不是继续依赖 `MONTHLY_BUDGET_LIMIT` 常量：

- `defaultBudget`
  - 类型：数字
  - 含义：默认月预算
- `monthlyBudgets`
  - 类型：按 `monthKey` 索引的预算覆盖项集合
  - 含义：某个月显式设置的预算

### 实际预算解析规则

统一通过一个解析函数获取某个月的实际预算：

1. 如果该月存在月覆盖预算，使用该值。
2. 否则如果存在默认预算，使用默认预算。
3. 否则使用内置兜底值 `2000`。

这样即使用户是从旧版本升级，也不会因为预算配置缺失导致页面报错或统计异常。

### 存储位置

预算配置持久化到 SQLite，和账单、分类一样保存在本地数据库中。

建议新增独立预算配置表，而不是把预算塞进分类或账单表中。原因：

- 预算是账本级配置，不属于单笔记录。
- 预算配置不应该跟随分类结构耦合。
- 独立表更方便备份导出与导入恢复。

建议结构：

- `budget_settings`
  - `scope`: `default` 或 `month`
  - `month_key`: 对于默认预算为空，对于月覆盖预算存具体月份
  - `amount`
  - `updated_at`

约束：

- 只能存在一条 `scope = default` 的默认预算记录。
- 每个 `month_key` 最多存在一条月覆盖预算记录。

## 汇总层改造

### 现状问题

当前 [ledgerSummary.ts](/D:/mcat/src/lib/ledgerSummary.ts) 里所有预算计算都基于固定常量：

- `selectedBudget`
- `monthlyBudgetRows`
- `totalOverspend`
- `totalRemaining`
- `netBudgetBalance`
- `averageMonthlyOverspend`
- 趋势图预算基准

这意味着一旦预算可配置，现有汇总逻辑必须改成按月读取预算，否则显示口径一定会错。

### 新汇总规则

汇总层增加预算配置输入，并对每个月分别计算：

- `budgetLimit`
- `remaining`
- `overspend`
- `utilizationRate`
- `isOverBudget`

其中：

- 选中月份预算快照使用 `selectedMonth` 对应预算。
- 跨月预算状态列表按每个月自己的预算计算。
- `totalOverspend` 为“每个月超支部分之和”。
- `totalRemaining` 为“每个月结余部分之和”。
- `netBudgetBalance` 为 `totalRemaining - totalOverspend`。
- `averageMonthlyOverspend` 继续按“总超支 / 有数据月份数”计算，但总超支口径改为动态预算口径。

### 预算快照结构

`BudgetSnapshot` 和 `BudgetMonthRow` 需要补充 `budgetLimit` 字段，避免展示层还要额外传另一份预算值。

这样好处是：

- 概览页预算条直接使用 `summary.selectedBudget.budgetLimit`
- 趋势页每月预算状态直接读每行的 `budgetLimit`
- 预算展示和预算计算来源完全一致

## 展示层同步范围

### 概览页

概览页需要同步更新以下内容：

- Hero 卡片预算条的预算上限值
- `本月预算状态`
- `累计超支`
- `综合超支情况`

其中预算条不再从全局常量读取，而是读取选中月份的实际预算。

### 趋势页

趋势页需要同步更新以下内容：

- 月趋势图预算基准线
- `每月预算状态`
- `超支月份排名`

月趋势图里的预算线以“当前选中月份预算”为基准，这样和当前月份切换保持一致。月预算状态与超支排名则按各自月份预算计算。

### 文案同步

所有带有固定“预算 2000”心智的展示都要改成动态文案，例如：

- `预算 {formatCurrency(actualBudget)}`
- 月度状态卡中的预算含义以该月预算为准

README 中涉及固定预算 `2000` 的对外说明也要改成“支持自定义预算设置”，避免文档与产品不一致。

## 统一入口设计

### 入口命名

记账页顶部原本分开的：

- `账单备份与导入`
- `分类管理`

统一整合为一个入口：

- `账本设置`

推荐原因：

- 范围足够大，能自然覆盖预算、分类、备份三类配置。
- 比“数据管理”更贴近用户理解。
- 比“高级设置”更不带技术感。

### 入口结构

点击 `账本设置` 后，打开统一浮窗，内部包含三个区块：

1. `预算设置`
2. `分类管理`
3. `备份与导入`

### 组织方式

- `预算设置` 作为浮窗顶部的首个模块。
- `分类管理` 继续使用现有独立弹窗能力，从统一入口内二次进入。
- `备份与导入` 继续复用现有卡片样式和操作逻辑。

这样做的原因是分类管理交互已经比较复杂，不适合直接平铺到统一入口主视图中；预算设置和备份导入则适合作为一级设置内容展示。

## 预算设置交互

### 预算设置卡片内容

预算设置卡片包含：

- 当前默认预算
- 修改默认预算入口
- 当前录入月份的预算状态说明
- 为当前月份设置单独预算的入口
- 若当前月份已存在覆盖预算，则显示“恢复默认预算”

### 预算编辑方式

采用与当前产品风格一致的浮层编辑方式：

- 点击“修改默认预算”时，弹出一个小型输入浮层
- 点击“为本月单独设置预算”时，弹出另一个输入浮层
- 输入框保持在键盘上方，沿用当前已验证过的悬浮输入风格

### 预算规则提示

为了让用户理解“默认预算”和“月覆盖预算”的关系，需要在预算设置卡片中明确提示：

- 当月未单独设置时，自动跟随默认预算
- 当月设置单独预算后，仅该月受影响
- 恢复默认预算会删除该月覆盖配置

## 备份与导入联动

### 导出

备份导出文件新增预算配置字段。导出内容应包括：

- 账单记录
- 分类定义
- 默认预算
- 月覆盖预算列表

### 导入

导入恢复同样要处理预算配置：

- 合并导入：
  - 账单、分类按现有规则处理
  - 预算配置建议采用“导入文件覆盖本地预算配置”的策略
- 覆盖恢复：
  - 清空本地账单、分类、预算配置
  - 以备份文件内容完整恢复

本次推荐在 UI 上明确说明预算配置也会参与恢复。

### 备份格式版本

备份 schema version 需要升级，避免旧版解析逻辑把新字段误认为无效，或新版导入旧文件时无法区分预算配置是否缺失。

新版解析需要兼容两类情况：

- 新版备份：包含预算配置
- 旧版备份：不包含预算配置，此时回退到默认预算兜底逻辑

## 数据流

整体数据流调整为：

1. 启动时读取账单、分类、预算配置
2. 汇总层接收账单 + 分类 + 预算配置
3. 汇总层按月解析预算后生成统一 `LedgerSummary`
4. 概览页与趋势页只消费 `LedgerSummary`
5. 记账页顶部的 `账本设置` 负责修改预算、分类和备份入口
6. 预算变更后刷新数据，概览和趋势自动同步

这保证了预算相关页面不会各自算一套口径。

## 错误处理

- 输入预算为空、非数字或小于等于 0 时，不允许保存。
- 如果用户删除了某月覆盖预算，该月立即恢复跟随默认预算。
- 若预算配置读取失败，不阻塞账单展示，但应回退到兜底预算并显示错误提示。
- 导入旧版本备份文件时，若文件中没有预算配置，不报错，直接走默认预算兜底。

## 测试策略

### 单元测试

需要补充以下测试：

- 默认预算生效
- 月覆盖预算优先于默认预算
- 删除月覆盖预算后恢复默认预算
- 不同月份使用不同预算时，`totalOverspend` / `totalRemaining` / `netBudgetBalance` 计算正确
- 备份导出包含预算配置
- 导入恢复后预算配置正确落库

### 集成验证

重点验证以下交互：

- 修改默认预算后，概览页和趋势页同步更新
- 为单独月份设置预算后，只影响该月
- `账本设置` 能正常进入预算、分类、备份三个功能
- 备份导出后再导入，预算配置不丢失
- 真机构建安装后，预算设置和入口组织正常工作

## 风险与约束

### Android 原生目录不入 Git

当前仓库的 `/android` 目录在 `.gitignore` 中，不会进入版本控制。这意味着本次实现应尽量落在 Expo 配置、JS/TS 逻辑和 SQLite 数据层中，避免把关键能力依赖在无法提交的原生源码修改上。

### 汇总口径回归风险

预算计算改动会直接影响概览页和趋势页多个指标，一旦有一处仍然使用旧常量，页面就会出现口径不一致。因此实现时应优先改造汇总层，再改展示层，而不是在多个页面分别补丁式调整。

## 实施边界

本次只做以下能力闭环：

- 默认预算设置
- 月覆盖预算设置
- 预算展示同步
- 统一入口整合
- 预算配置备份导入

不在本次范围内的内容包括：

- 分类预算
- 年预算或周预算
- 预算提醒
- 云端同步

## 结论

本次应以“账本设置”为统一入口，新增“默认预算 + 按月覆盖”的预算体系，并把预算配置纳入 SQLite 持久化、汇总层动态计算和备份导入恢复。这样既能满足用户对预算灵活性的真实需求，也能让概览页、趋势页和管理入口继续维持清晰一致的产品结构。
