# 连续记账体验优化设计

## 背景

当前“记账”页的录入流更偏单次提交：保存后会直接切到“概览”，用户如果要连续录入多笔支出，就必须再次回到“记账”页并重新定位输入位置。同时，保存按钮位于表单底部，软键盘弹起后容易被遮挡，导致保存动作需要先收起键盘。概览页中的“超支月份数”也只反映发生次数，不能表达整体预算健康状况。

## 目标

- 保存成功后优先服务连续录入，而不是强制跳回“概览”。
- 保留月份上下文，只清空下一笔重新输入时真正需要重填的字段。
- 让保存动作在键盘弹起时仍然易于点击。
- 将概览页指标从“超支月份数”调整为更直接的综合超支结果。

## 非目标

- 不修改数据库结构。
- 不引入日期维度，仍然只按月份记账。
- 不新增新的导航页或复杂手势交互。

## 交互方案

### 记账页连续录入

保存成功后不再统一跳去“概览”，而是根据当前大类所处位置决定下一步：

- 若当前大类不是最后一个，则停留在“记账”页并自动切到 `CATEGORY_DEFINITIONS` 中的下一个大类。
- 若当前大类已经是最后一个 `其他`，则本次保存完成后切到“概览”页。

保存后的字段处理规则：

- `monthKey` 保持当前值不变。
- `amount` 清空。
- `note` 清空。
- `category` 切到下一个大类，或在最后一个大类时不再继续停留。
- `subcategory` 跟随新的大类默认回到该大类的第一个子类；若新大类没有子类，则为空。

这样可以让连续录入在大类维度上自然向后推进，同时符合用户指定的“最后一个大类后跳概览”。

### 保存按钮可点击性

保存按钮从滚动内容底部挪到固定底部操作区：

- 表单主体继续使用 `ScrollView`。
- 保存按钮独立为底部悬浮操作条，固定在 tab bar 上方。
- 记账页外层继续使用 `KeyboardAvoidingView`，同时给内容区预留足够底部留白，避免最后一个输入框被按钮压住。

这个方案的重点不是单纯“往上挪一点”，而是让保存按钮脱离长表单滚动流，在键盘弹出时仍保留稳定的点击区域。

### 概览页综合超支指标

把概览页指标 `超支月份数` 替换为 `综合超支情况`。

统计口径：

- `总超支 = 各月份 max(月支出 - 预算, 0) 的总和`
- `总结余 = 各月份 max(预算 - 月支出, 0) 的总和`
- `净值 = 总超支 - 总结余`

展示规则：

- `净值 > 0` 时显示 `净超支 X`
- `净值 <= 0` 时显示 `净结余 X`

这比“超支月份数”更能反映整体预算健康，因为它同时考虑了超支和结余的抵消关系。

## 数据与代码设计

### 汇总层

在 `src/lib/ledgerSummary.ts` 中扩展预算汇总结果：

- 保留现有 `totalOverspend`、`overspendMonthCount` 等字段。
- 新增 `totalRemaining`，表示所有月份总结余。
- 新增 `netBudgetBalance`，值为 `totalRemaining - totalOverspend`。当它为负数时代表净超支，为正数或零时代表净结余。

采用 `remaining - overspend` 的方向可以让展示层更直接判断正负语义，同时不破坏现有 `BudgetSnapshot` 的结构。

### 记账页表单

在 `src/components/ExpenseForm.tsx` 中增加一个“保存后的下一步”计算逻辑：

- 把“根据当前大类找到下一个大类”的逻辑抽成纯函数，便于测试。
- `onSubmit` 从仅负责持久化，扩展为返回保存后的导航意图，或接收足够信息让表单自行决定后续状态。

推荐做法是让父组件 `App.tsx` 不再在保存后无条件切换 tab，改为由表单在成功保存后按规则决定：

- 连续录入场景：保持在 `add`
- 最后大类完成：切回 `overview`

这样可以把“表单状态重置”和“后续导航”放在同一个交互上下文里处理，减少父子组件来回同步。

## 测试策略

优先补纯逻辑测试：

- `ledgerSummary` 增加综合超支口径测试，覆盖超支与结余抵消后的结果。
- 为“下一个大类”逻辑增加测试，覆盖中间大类与最后大类两种情况。

UI 侧不额外引入新的测试框架。本轮通过：

- TypeScript 编译校验
- Vitest 纯函数测试
- 必要时对关键布局属性做人工代码检查

主要改动文件：
