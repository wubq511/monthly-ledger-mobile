# 分类拖拽排序与连续录入推进设计

## 背景

当前分类管理已经支持大类与细分的新增、改名、删除，也支持大类通过“长按选中，再点击目标项前插”的方式调整顺序。

这套交互已经能完成排序，但不够直接；同时连续录入只会在大类之间推进，没有把细分顺序纳入流程，导致“有细分的大类”录入效率仍然偏低。

本轮设计只解决这两个问题：

- 把顺序调整改成真正的长按拖动
- 把保存后的自动推进规则改成“先细分，后大类”

## 目标

- 大类支持长按拖动排序
- 细分也支持长按拖动排序
- 大类和细分的拖拽顺序都持久化到 SQLite
- 保存后自动跳转遵循当前真实排序，而不是只看大类
- 如果当前大类存在细分，优先推进到下一个细分
- 当前细分结束后，再推进到下一个大类

## 非目标

- 本轮不修改分类的增删改权限
- 本轮不修改备份文件结构
- 本轮不做跨大类拖动细分
- 本轮不新增第三层分类

## 核心行为

### 拖拽排序

#### 大类

- 用户长按某个大类卡片后进入拖动态
- 手指上下拖动，卡片跟随移动
- 松手后，按最终位置更新大类顺序
- 新顺序写入 `categories.sort_order`

#### 细分

- 用户长按某个细分行后进入拖动态
- 只允许在所属大类内部排序
- 松手后，按最终位置更新该大类下细分顺序
- 新顺序写入 `subcategories.sort_order`

### 连续录入推进

保存成功后，推进逻辑改为：

1. 找到当前大类在当前排序中的位置
2. 如果当前大类有细分：
   - 若当前细分后面还有下一项，跳到下一个细分
   - 若当前细分已经是最后一个，进入下一大类
3. 如果当前大类没有细分，直接进入下一大类
4. 进入下一大类时：
   - 若该大类有细分，默认选中它的第一个细分
   - 若没有细分，则清空细分
5. 如果当前已经是最后一个大类，且没有更多细分，则返回概览页

### 异常与兼容

- 如果当前选中的大类或细分在保存前刚被删除，推进逻辑直接回退到“当前排序下的第一个可用位置”
- 历史账单兼容规则保持不变：
  - 删除分类不会删除历史账单
  - 改名会同步改历史账单
- 备份恢复不需要改 schema，因为当前 `CategoryRecord` 已包含 `sortOrder`

## 数据设计

### 数据库

已有字段足够承载本轮需求：

- `categories.sort_order`
- `subcategories.sort_order`

需要补的只是数据库更新方法：

- 批量更新大类顺序
- 批量更新指定大类下的细分顺序

### 内存结构

运行时仍然使用 `CategoryRecord[]`：

- 大类数组顺序代表大类真实顺序
- `category.subcategories` 顺序代表该大类下细分真实顺序

## 交互设计

### 分类管理弹窗

- 保留当前的弹窗结构、按钮分布和编辑/删除操作
- 删除“长按选中、点击目标项”的提示和状态
- 每个大类卡片显示拖拽手柄，用来暗示可拖动
- 每个细分行也显示拖拽手柄
- 拖拽时，当前项使用轻微提起态，目标区域自然让位

拖拽体验要求改成对齐主流移动端列表：

- 长按手柄后，只有当前项浮起并跟手移动
- 列表中的其他项实时让位，而不是松手后再跳位
- 分类管理弹窗本身不应在拖拽过程中跟着滚动位移
- 动效使用平滑的缩放、阴影和占位过渡
- 细分列表也要具备相同体验，而不是“可拖但生硬”

## 实现边界

### 推荐实现方式

使用成熟拖拽栈实现，而不是继续自研手势逻辑：

- `react-native-gesture-handler`
- `react-native-reanimated`
- `react-native-draggable-flatlist`

原因：

- 当前自研方案无法可靠接管滚动与拖动冲突
- 主流拖拽体验需要悬浮项、占位动画、相邻实时交换和更稳定的手势仲裁
- 这些都是成熟拖拽库已解决的问题，继续补现有方案只会反复返工

### 列表组织

- 大类使用真正的拖拽列表组件渲染
- 细分也使用真正的拖拽列表组件渲染
- UI 上仍然保持“一个大类卡片，内部带细分列表”的结构
- 拖拽的状态管理由组件本地维护，用于提供即时的视觉反馈
- 松手后再调用数据库更新接口持久化顺序

### 依赖配置

- 入口文件需要接入 `react-native-gesture-handler`
- 根组件需要放入 `GestureHandlerRootView`
- 项目新增 `babel.config.js`，为 Reanimated 启用 Babel 插件

### 组件拆分

- `CategoryManagerModal.tsx`
  - 使用拖拽列表库实现大类与细分的平滑拖拽 UI
- `categoryReorder.ts`
  - 提供通用数组重排工具与本地乐观重排 helper
- `database.ts`
  - 提供大类/细分顺序持久化接口
- `expenseFormFlow.ts`
  - 提供“先细分、后大类”的下一步算法
- `ExpenseForm.tsx`
  - 在保存成功后使用新的推进结果更新表单状态
- `index.ts`
  - 注册手势环境
- `babel.config.js`
  - 启用 Reanimated Babel 插件

## 测试要求

至少覆盖：

- 大类顺序更新后，读取结果按新顺序返回
- 细分顺序更新后，读取结果按新顺序返回
- 推进逻辑在“同大类内还有下一个细分”时跳到下一个细分
- 推进逻辑在“细分结束”时跳到下一个大类
- 推进逻辑在“无细分大类”时直接跳到下一个大类
- 推进逻辑在最后一个位置时返回概览页
- 通用重排 helper 在头部、尾部、中间拖动时都返回正确顺序
- 本地乐观重排 helper 能正确更新嵌套大类/细分树

## 验收标准

- 用户可以在分类管理弹窗中长按拖动大类排序
- 用户可以在分类管理弹窗中长按拖动细分排序
- 拖动大类时，只有当前大类卡片浮起移动，浮窗本身不跟着误滚动
- 拖动细分时，列表实时让位，视觉上是连续、平滑的交换过程
- 重开应用后，大类与细分顺序保持不变
- 连续录入在有细分时先跳下一个细分
- 连续录入在细分结束后跳下一个大类
- 最后一项录入完成后仍返回概览页
